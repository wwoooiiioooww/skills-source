# friction-minimal-ux — 摩擦最小パーソナルアプリの設計思想

対象ユーザー像: Quantified Self系の意識が高い人ではなく、**記録という行為に興味がなく、ウェアラブルも買わず、忙しさに生活が浸食されている人**。この前提を外すと市販の「毎日対話してくる・ウェアラブル前提・クラウド前提」設計に引っ張られ、使われなくなる。

## 絶対原則

1. **入力させたら負け。** 新機能がユーザーの入力手数を増やすなら、まず疑う。確認ダイアログ・必須入力欄・複数ステップのフォームは原則禁止。「1日の能動的入力は合計10秒以内」を定量上限とする。入力欄を1つ増やす前に「選択にできないか」「自動導出できないか」を必ず問う。
2. **観測が先、介入は後。** アプリは記録し可視化する。助言・通知での割り込みは週1回等の低頻度に限定。
3. **説教禁止・仮説提案型。** 一般論の押し付け（「バランスの良い食事を」）を出力しない。提案は「データのこの部分から、来週これを1つだけ試してみては」の形式で、**常に1つだけ**（2つ以上は選択コストで何もしなくなる）。
4. **折れても責めない。** 欠損は欠損のまま扱い、埋めるよう促さない。ストリークをUI前面に出さない（途切れた瞬間の離脱トリガーになる）。リマインドは1回通知して反応がなければ終わり。追撃しない。
5. **ローカル完結・プライバシー最優先。** 生体・行動データは端末内に閉じる。外部送信は明示的に必要な処理に限定し、集計値のみ送る（生ログを送らない）。
6. **生活様式を矯正しない。** 「1日1〜2食」等、本人が意図的に選んだパターンを正論で書き換えない。
7. **時間の定義は生活リズムに合わせる。** 0時〜24時の暦日を無条件採用しない。深夜型には「論理日（例: 朝4時区切り）」を設計する。
8. **UIの方向性は一貫させる。** 「悪い→良い」は常に同じ方向（例: 左→右）。軸ごとに逆転させない（食事評価だけ逆順だった実例あり）。
9. **精度より運用可能性。** スライダー等の繊細な操作を避け、タップで完結する離散的選択肢（4段階セグメント等）にする。

## 設計チェックリスト（摩擦監査）

- [ ] この機能は何タップ必要か、実際に数えたか。3タップ超なら簡略化を検討
- [ ] 必須と任意を分離し、任意にはスキップ導線があるか。数秒操作しなければ自動で消える設計にできないか
- [ ] 確認ダイアログを本当に必要とするか（不可逆操作以外は不要）。Undoで代替できないか（確認より取り消しの方が低摩擦）
- [ ] 同じ情報を2つの軸で重複して聞いていないか。既存軸と直交する情報を2段階目・任意に配置したか
- [ ] デフォルト選択肢が開発者個人の生活に偏っていないか。デフォルトはMECEな「種類」軸にし、個人の好みは追加設定に逃がす
- [ ] 深夜をまたぐ操作が意図した論理日に帰属するか、境界値でテストしたか
- [ ] 「今日」「今週」の時間定義が全画面で統一されているか
- [ ] 通知・レポート・エラーの文言が説教・一般論になっていないか

## 良い例／悪い例（代表）

- 睡眠: ❌毎晩2つの必須フォーム → ⭕任意の「おやすみ」1タップ＋操作間隔からのタップフリー自動推定＋手動修正導線
- 食事: ❌カロリー・PFC入力 → ⭕健康度3段階1タップ。種類・店は完全任意で自動消滅
- 通知: ❌途切れたら毎日再送 → ⭕1回だけ、無視されたら次周期まで待つ。責める文言ゼロ

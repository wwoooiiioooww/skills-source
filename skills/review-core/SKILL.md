# review-core — 全プロジェクト共通のメタ原則

対象: Shotaの個人開発プロジェクト全般（Studyきち、ヘルスきち、地下鉄クエスト、AI教育カリキュラム、および今後の新規アプリ）。
本Skillはドメインを問わない「プロセスと品質の土台」。ドメイン固有の観点は各専用Skillに委ねる。

## プロセス原則（作業の進め方）

1. **計画先行（Plan-first）**: 実装・執筆に着手する前に、必ず計画（何を・どの順で・どこを変更するか・DoD）を提示し、承認を得る。高価な手戻りを安価なチェックポイントに置き換えるのが目的。
2. **討論→合意→実装**: 設計判断は「選択肢＋推奨＋理由」の形で提示して合意を得てから作る。特に運用に効く判断（報酬・承認・データ境界など）は先に握る。
3. **完成の定義（DoD）を明文化**: 「何ができたら終わりか」を着手前に機械判定可能な形で書く。DoDが曖昧なままの着手は禁止。
4. **段階納品**: 大きな機能はa/b等に分割し、各ステップで自動テスト全パスしてから渡す。データを貯める側（入力）を先、使う側（レポート・グラフ）を後。
5. **スコープ厳守**: 依頼されていない機能追加・リファクタリングを混ぜない。気づいた改善点は「提案」として分離して報告する。
6. **検証状態の明記**: ビルド・実機確認が未実施の場合、その旨をコード内コメントと納品報告に明記し、次の担当（人間またはAI）が優先的に検証すべき箇所を判別できるようにする。

## データ・互換性の原則

7. **スキーマ変更には必ずmigrate**: 旧バージョンデータの読込経路とバックアップJSONのimport経路の両方に同じmigrateを通す。ユーザーのデータを壊す変更は一切禁止。
8. **カスタマイズ済み項目を上書きしない**: デフォルト値の更新時は「初期値のまま未編集か」を判定し、未編集の場合のみ新デフォルトへ置換。ユーザーが編集した項目は保持する。
9. **推定できないなら null**: 精度の低い推測値を出すくらいなら「わからない」と正直に扱う。欠損を0やダミー値で埋めない。
10. **時系列データは描画前に日付ソート**: 配列の登録順に依存しない。保存時とmigrate時の両方でソートする。

## AI API連携の型（Gemini / Claude API等）

- **maxOutputTokensは4096以上**: 思考型モデルは内部思考もトークン上限にカウントされる。ケチると本文が尻切れになる（実例あり: 600設定で毎回失敗）。
- **空文字フォールバック禁止**: モデル名・設定値が空/未設定のときは必ずコード側のデフォルト定数に解決してからAPIに渡す。
- **キー保存は「保存→即時検証→結果表示」の3点セット**: 保存だけして無言で失敗する経路を作らない。
- **応答解析は思考パーツ・コードフェンス除去後に行う**。
- **エラーは展開可能な詳細付きで残す**: 「時間をおいて再試行」で終わらせず、HTTPステータスやレスポンス断片を展開表示できる形にし、原因の切り分けを可能にする。
- **リトライは自動追撃せずユーザー操作起点にする**。

## 配布・更新の原則

- **キャッシュを伴う配布（Service Worker等）は更新のたびにキャッシュキーをバンプ**し、画面上のバージョン表示（vN）とセットで上げる。
- **プラットフォーム規約はアプリロジックより優先**（例: GitHub Pagesのルート表示は index.html のみ）。
- **配布物はフォルダ非依存（フラット構成）を基本とする**: 手動アップロードで階層が平坦化されて壊れた実例があるため。

## 機械的検証の原則（目視に頼らない）

- **構文チェックを実装のたびに必ず実施**: `node --check`、または`<script>`抽出＋`new Function()`パース。目視だけで済ませない。
- **文字列置換の前に実ファイルをgrepで照合**: 思い込みでassertを書かない。**中断から再開するときは、どの編集が適用済みかを必ずgrepで棚卸ししてから続きをやる。**
- **AI生成時に紛れ込みやすい誤字をgrepで確認**（例:「AI」→「AV」化）。
- **既存ファイルへのcreate_fileは失敗する**: 事前に存在確認し、必要なら削除してから作成する。
- **テストは「動作確認」ではなく「過去に発生した実際の不具合の再発防止」を主目的に書く**。失敗パターンは発生のたびにテストケース化する。

## レビュー時のチェックリスト（成果物をGoにする前に）

- [ ] DoDを満たしているか（計画時に合意したDoDと突き合わせる）
- [ ] スコープ外の変更が混入していないか（diffを確認）
- [ ] migrate・後方互換の考慮があるか（スキーマ/デフォルト値変更時）
- [ ] キャッシュキー・バージョン表示の同時更新（配布物の場合）
- [ ] 構文チェック・自動テストの結果が報告に含まれているか
- [ ] エラーハンドリングが「原因の切り分け可能」な形か
- [ ] 未検証箇所が明記されているか
- [ ] 報告は結論ファーストか（何ができた→何が未了→判断してほしい点）

## 報告の型

バグ修正の報告は「直しました」で終わらせず、原因を平易に説明する（例:「思考トークンも上限にカウントされるため尻切れが起きた」）。信頼と次の要望の質につながる。
